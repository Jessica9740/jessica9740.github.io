# Claude API 토큰 절약하는 7가지 실전 방법

Claude API를 본격적으로 서비스에 도입하면 가장 먼저 마주치는 현실은 **빠르게 늘어나는 토큰 비용**입니다. Anthropic의 가격 정책상 Claude 3.5 Sonnet은 입력 토큰 100만 개당 $3, 출력 토큰 100만 개당 $15가 부과되며, Opus 모델은 그보다 약 5배 비쌉니다. 사용량이 늘수록 작은 비효율이 큰 비용으로 돌아옵니다.

이 글에서는 단순한 "프롬프트를 짧게 쓰세요" 수준을 넘어, **실제로 측정 가능한 효과를 내는 토큰 절약 전략 7가지**를 소개합니다.

## 1. 프롬프트 캐싱(Prompt Caching) 적극 활용하기

2024년 8월부터 Anthropic이 정식 제공 중인 **프롬프트 캐싱**은 가장 큰 비용 절감 수단입니다. 시스템 프롬프트, 긴 문서, 예시(few-shot examples) 등 반복되는 부분을 캐시에 저장하면:

- **캐시 쓰기**: 기본 입력 가격의 1.25배
- **캐시 읽기**: 기본 입력 가격의 **0.1배 (90% 할인)**

예를 들어 10만 토큰짜리 매뉴얼을 매번 컨텍스트에 넣어 질의한다면, 캐싱 없이는 매 호출마다 $0.30이 들지만 캐싱을 사용하면 두 번째 호출부터 $0.03으로 줄어듭니다.

```python
response = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    system=[
        {
            "type": "text",
            "text": "긴 시스템 프롬프트...",
            "cache_control": {"type": "ephemeral"}
        }
    ],
    messages=[...]
)
```

캐시 수명은 기본 5분이며, 같은 프롬프트가 반복 호출될 때마다 갱신됩니다. **챗봇, 문서 Q&A, 코드 리뷰 도구**처럼 동일한 컨텍스트를 반복 사용하는 시나리오에서 효과가 극대화됩니다.

## 2. 작업에 맞는 모델 선택: Haiku를 무시하지 마세요

많은 개발자들이 무조건 Sonnet이나 Opus를 사용하지만, 실제 작업의 60~70%는 **Haiku**로 충분히 처리됩니다.

| 모델 | 입력 비용 (1M 토큰) | 출력 비용 (1M 토큰) |
|------|------------------|------------------|
| Claude 3.5 Haiku | $0.80 | $4.00 |
| Claude 3.5 Sonnet | $3.00 | $15.00 |
| Claude 3 Opus | $15.00 | $75.00 |

**모델 라우팅 전략**을 적용하면 좋습니다:
- 분류, 요약, 단순 추출 → **Haiku**
- 복잡한 추론, 코드 생성 → **Sonnet**
- 고난도 분석, 창작 → **Opus**

Haiku로 1차 처리 후 신뢰도가 낮은 경우만 Sonnet으로 에스컬레이션하는 패턴은 비용을 평균 70% 줄입니다.

## 3. 출력 토큰을 적극적으로 제한하기

출력 토큰은 입력보다 **5배 비쌉니다**. 따라서 출력 길이 제어가 비용에 미치는 영향은 매우 큽니다.

- `max_tokens` 파라미터를 실제 필요한 만큼만 설정
- 프롬프트에 "100단어 이내로 답변", "JSON만 출력하고 설명 생략" 같은 명시적 지시
- 불필요한 인사말, 자기 설명("물론이죠! 도와드리겠습니다...")을 차단

특히 구조화된 출력이 필요할 때는 **JSON 스키마**를 명확히 제시해 군더더기를 없애는 것이 중요합니다.

## 4. 컨텍스트 윈도우 관리: 슬라이딩 vs 요약

긴 대화에서 모든 히스토리를 매번 전송하면 토큰 비용이 **선형적으로 증가**합니다. 대신:

**(1) 슬라이딩 윈도우**: 최근 N개 턴만 유지
**(2) 요약 압축**: 오래된 대화는 LLM으로 요약 후 그 요약본만 전달
**(3) 하이브리드**: 최근 5턴은 원본 + 그 이전은 요약본

실무 데이터를 보면 100턴 이상의 긴 대화에서 요약 압축 방식이 토큰 사용량을 **80% 이상 감소**시키는 것으로 확인됩니다. 단, 요약 자체에도 토큰이 들어가므로 일정 길이를 넘어설 때만 트리거하는 것이 합리적입니다.

## 5. 프롬프트 자체를 다이어트하기

프롬프트 엔지니어링에서 흔히 간과되는 부분이 **불필요한 장황함**입니다. 다음 원칙을 적용해보세요:

- **마크다운 대신 간결한 구조**: `### 매우 중요한 지시사항 ###` 같은 장식 제거
- **예시(few-shot)는 꼭 필요한 만큼만**: 3개로 충분한데 10개를 넣고 있지 않은지 점검
- **중복 지시 제거**: "정확하게, 신중하게, 꼼꼼하게 답변하세요" → "정확하게 답변하세요"
- **약어/축약 활용**: 토큰화 관점에서 "예를 들면" 보다 "예:"가 효율적

Anthropic의 공식 토크나이저 또는 `client.messages.count_tokens()` API로 사전에 토큰 수를 측정하는 습관을 들이면 좋습니다.

## 6. RAG로 컨텍스트 주입 최적화하기

전체 문서를 통째로 컨텍스트에 넣지 말고, **RAG(Retrieval-Augmented Generation)** 로 관련 청크만 추출해 주입하세요.

- 문서 전체 50,000 토큰을 매번 전송 → $0.15/요청
- 관련 청크 2,000 토큰만 전송 → $0.006/요청

**25배 절감**입니다. 벡터 DB(Pinecone, Weaviate, pgvector 등)와 결합하면 정확도도 오히려 향상됩니다. 검색 품질이 충분하다면 컨텍스트가 짧을수록 모델의 응답 품질도 좋아지는 경향이 있습니다.

## 7. 배치 API 활용 (Message Batches API)

실시간성이 필요 없는 작업이라면 **Batches API**를 활용하세요. 24시간 내 처리 보장 조건으로 **모든 토큰 가격이 50% 할인**됩니다.

적합한 사용 사례:
- 대량 문서 분류/태깅
- 데이터 정제 및 라벨링
- 야간 일괄 분석 작업
- 임베딩 생성 전처리

```python
batch = client.messages.batches.create(
    requests=[
        {"custom_id": "task-1", "params": {...}},
        {"custom_id": "task-2", "params": {...}},
    ]
)
```

수만 건의 작업을 처리해야 하는 백엔드 파이프라인이라면 단순히 Batches API로 전환하는 것만으로도 비용이 정확히 절반이 됩니다.

## 모니터링과 알림: 절약의 출발점

마지막으로, 이 모든 전략을 적용하기 전에 **현재 어디서 토큰이 새고 있는지** 측정해야 합니다.

- Anthropic Console의 **Usage 대시보드**로 모델별, 일자별 사용량 추적
- 응답 객체의 `usage` 필드(`input_tokens`, `output_tokens`, `cache_read_input_tokens`)를 로깅
- 비정상적인 사용량 급증에 대한 알림 설정

**측정되지 않는 것은 최적화할 수 없습니다.**

## 마치며

지금까지 살펴본 7가지 전략을 정리하면 다음과 같습니다:

1. 프롬프트 캐싱 (최대 90% 할인)
2. 적절한 모델 선택 (Haiku 활용)
3. 출력 토큰 제한
4. 대화 히스토리 압축
5. 프롬프트 다이어트
6. RAG 기반 컨텍스트 최적화
7. Batches API 활용 (50% 할인)

이 중 **두세 가지만 제대로 적용해도 월 API 비용을 50% 이상 절감**할 수 있습니다. 특히 캐싱과 모델 라우팅, RAG 조합은 거의 모든 LLM 애플리케이션에 적용 가능한 황금 조합입니다.

비용 절감은 단순히 돈을 아끼는 문제가 아니라, **더 많은 사용자에게 AI 기능을 제공할 수 있는 여력**을 만드는 일입니다. 오늘 당장 자신의 프로젝트에서 한 가지 전략을 적용해보세요.
